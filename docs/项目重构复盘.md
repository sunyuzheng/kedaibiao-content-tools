# 项目重构复盘：脚本工程的常见范式问题

> 背景：对「课代表立正频道内容库」进行了一次从头到尾的审计与重构，
> 修复了元数据错误、清理了孤儿脚本、重新梳理了目录结构。
> 本文总结过程中发现的范式问题及其背后的原则。

---

## 一、用静态文件做「实时状态」

### 问题

项目用一个 CSV 文件（`视频与Transistor同步状态.csv`）记录本地视频与 Transistor 的同步状态。
每次上传后需要手动重新生成，否则就是过期数据。实际上几乎从未被更新。

### 修法

删掉 CSV，改用实时查询脚本（`check_upload_candidates.py`、`check_transistor_sync.py`），
直接对比本地文件系统 + 远端 API，终端打印结果，零文件输出。

### 原则

**需要查的状态，就去查；不要缓存成文件。**

静态文件天然会腐烂。对于「现在有多少待上传视频」这种问题，正确答案永远在源头（本地 archive + 远端 API），
而不在一个上次生成时的快照里。
只有「生成成本很高、变化很慢」的数据才值得缓存成文件。
否则，文件的存在本身就是一种误导——它看起来像答案，但往往不是。

## 二、目录名称是一份合约

### 问题

`check_upload_candidates.py`（只读检查）和 `find_duplicates.py`（只读诊断）都放在 `tools/upload/` 里。
`batch_transcribe_*.py`（Whisper 转录）放在 `tools/generate/` 里，跟生成报告的脚本混在一起。

### 修法

- `check_upload_candidates.py`、`find_duplicates.py` 移入 `tools/check/`
- `batch_transcribe_*.py` 移入新建的 `tools/transcribe/`
- `tools/generate/` 整个删掉（清空后没有存在意义）

### 原则

**目录名是对里面所有内容的承诺。**

开发者看到 `tools/upload/` 时，心智模型是「这里的脚本会向 Transistor 写数据」。
一旦里面混入只读脚本，这个承诺就破了，以后找东西要靠记忆而不是结构。

好的目录划分标准：**按操作性质**（读 vs 写）而非按使用场景（「上传相关的都放这里」）。
`tools/check/` 里的每个脚本都只读；`tools/upload/` 里的每个脚本都会修改远端状态。
这让权限意识、审计、调试都变得更容易。

## 三、工作流重构后，旧脚本成了负债

### 问题

`generate_summary.py` 生成「下载总结.md」，是早期工作流的产物——当时需要手动 review 下载进度。
后来工作流重构，这个步骤被自动化替代，但脚本留了下来。
`generate_video_documents.py` 把 metadata + SRT 合并成 Markdown，是为内容写作准备的，
但内容写作方向已转变为使用新方法，这套输出物已无人消费。

两个脚本加上它们的输出物（`下载总结.md`、475 个 `视频文档/*.md`）持续占据认知空间，
让人误以为项目依赖它们。

### 修法

全部删除：两个脚本、输出目录、孤儿数据文件（`youtube_all_videos.txt`）。

### 原则

**没有消费者的脚本是负债，不是资产。**

脚本的价值来自它被调用。一个不再被任何工作流调用的脚本，
维护成本（理解、更新、避免误运行）大于价值。
工作流重构时，要同步清理为旧工作流服务的工具，而不是「留着说不定有用」。

判断标准很简单：**这个脚本上次被运行是什么时候？被谁调用？**
答不出来，就是孤儿。

## 四、生成产物没有被 gitignore

### 问题

多个脚本的输出文件（`下载总结.md`、`数据完整性报告.md`、`duplicates_report.json/.csv`）
没有加入 `.gitignore`，存在被意外提交的风险。

### 修法

所有脚本运行产物加入 `.gitignore`。

### 原则

**凡是可以被脚本重新生成的文件，都不应该进版本控制。**

规则很简单：如果文件的内容是由另一个文件（脚本）确定性地产生的，
那它就是派生物，派生物不进 git。进了 git 就会出现「这个文件是旧的还是新的」的困惑，
以及毫无意义的 diff 噪声。

写完一个新脚本，第一件事就是想：它会输出什么文件？这些文件加进 `.gitignore` 了吗？

## 五、文档与实际结构的漂移

### 问题

重构过程中多次出现文档指向错误路径的情况：
- 文档写 `tools/generate/generate_video_transistor_sync_csv.py`，文件已移走
- 文档写 `tools/check/check_upload_candidates.py`，文件实际在 `tools/upload/`
- Archive 结构图里还有 `youtube_all_videos.txt`，文件已删除

### 修法

每次移动/删除文件时，原子性地更新所有引用该路径的文档。

### 原则

**文件移动和文档更新是同一个操作，不是两个操作。**

文档漂移的根本原因是把「移文件」和「改文档」当成两件事，
结果第二件事常常被遗忘。
正确的心智模型是：移文件 = 移文件 + 更新所有引用。
可以用 grep 搜索旧路径来做完整性检查。

## 六、API 集成的静默失败

### 问题

给 Transistor 的 episode 设置 `image_url` 时，使用了 YouTube 的 `maxresdefault.jpg`。
API 返回 200 OK，但实际上缩略图没有被存储——因为早期视频的 `maxresdefault.jpg` 在 YouTube 上根本不存在（404），
Transistor 静默丢弃了无效 URL。

同样，尝试通过 `PATCH /v1/episodes/:id` 修改 `published_at` 和 `status`，
API 返回成功，但字段没有变化——因为这些字段只能通过 `/publish` 子端点修改。

### 修法

- 图片一律改用 `hqdefault.jpg`（对所有 YouTube 视频都存在）
- `published_at` 通过 `PATCH /v1/episodes/:id/publish` 修改

### 原则

**API 集成要显式验证副作用，不能信任 200 OK。**

「调用成功」≠「操作生效」。外部 API 有两种典型的静默失败模式：
1. 接受无效输入但不报错（Transistor 接受 404 图片 URL）
2. 正确端点之外的字段修改被静默忽略（`/episodes/:id` 上的 status 字段）

写完 API 集成代码，一定要回读数据验证字段确实变了。
第一次集成时写一个 verify 步骤；遇到异常时优先怀疑静默失败。

## 七、有序性依赖被隐式假设

### 问题

上传脚本起初没有强制上传顺序，导致最新视频被分配到最小的 EP 号。
`reorder_episodes_by_date.py` 只更新 Transistor 的 `number` 字段，不更新 title 文本中的前缀，
导致 EP 号和标题前缀（`E233. xxx`）对不上。

### 修法

- 上传时强制按日期升序（旧的先传）
- 重排 EP 号后，增加标题前缀修复步骤

### 原则

**依赖顺序的操作，要在代码层面强制顺序，而不是靠约定。**

「先传旧的」不能是文档里的一句话，要在代码里 `sort(key=date, ascending=True)`。
冗余存储（EP number 字段 + title 文本里的前缀）必然会漂移，
要么消除冗余，要么每次修改都同步更新所有冗余副本。
两个地方存同一个信息，就要有机制保证它们一致。

## 总结：七条原则

| # | 原则 |
|---|------|
| 1 | 需要查的状态，就去查；不要缓存成文件 |
| 2 | 目录名是对里面所有内容的承诺，按操作性质划分 |
| 3 | 没有消费者的脚本是负债，工作流重构时同步清理工具 |
| 4 | 凡是可以被重新生成的文件，都不进版本控制 |
| 5 | 文件移动和文档更新是同一个操作 |
| 6 | API 集成要显式验证副作用，不能信任 200 OK |
| 7 | 依赖顺序的操作，在代码层面强制，不靠约定 |
