# 项目重构复盘

> 时间：2026-03-29
> 起点：Transistor 上有 450 个 episode，但元数据一团糟，本地项目结构混乱
> 终点：数据干净、结构清晰、文档可信、代码上了 GitHub

---

## 起始状态

| 问题 | 严重程度 |
|------|---------|
| 所有 450 个 episode 的 `published_at` 是批量上传时间，不是 YouTube 发布时间 | 高——podcast feed 排序完全错误 |
| EP441-450 缺少缩略图；更早的部分 episode 也没有 | 中 |
| 233 个 episode 标题前缀（`EP444_`）与实际 EP 号（441）对不上 | 中 |
| 对账方式依赖一个长期不更新的 CSV 文件 | 中 |
| 项目文档有 14K 字的 README，自写 API 手册可能过时 | 低 |
| 脚本放错目录、孤儿脚本、生成产物未 gitignore | 低 |
| API key 硬编码在三个脚本里 | 低（未公开）→ 上 GitHub 前变高 |

## 第一阶段：Transistor 元数据大修

### 发生了什么

**缩略图问题**：EP441-450 整批缺图。排查后发现，这批是用 `maxresdefault.jpg` URL 设置的——API 返回 200 OK，但 Transistor 静默丢弃了无效 URL，因为早期 YouTube 视频根本没有 `maxresdefault.jpg`（YouTube 只对新视频生成高清封面）。换成 `hqdefault.jpg` 后全部正常。

**`published_at` 问题**：所有 450 个 episode 的发布时间都是批量上传时的机器时间，而且 EP441-450 还是倒序的（最新视频反而时间最早），导致播客客户端把最新内容排在最前面。

**修复过程**：
1. 写了 `fix_episode_metadata.py`，从本地 archive 文件夹名提取每个视频的 YouTube 发布日期
2. 通过 `PATCH /v1/episodes/:id/publish` 将所有 450 个 episode 的 `published_at` 设为正确的 YouTube 日期（同一天的视频按分钟错开，保证顺序唯一）
3. 同步修复缩略图：跳过已有图片的 episode，只填补缺失的

**结果**：450 个日期修正，122 个缩略图补齐

### 学到了什么

**200 OK 不等于操作生效。** Transistor 对两类操作存在静默失败：

1. 接受无效 URL 但不存储（`maxresdefault.jpg` 404 了，API 不报错）
2. 部分字段只能通过子端点修改——`published_at` 和 `status` 必须走 `PATCH /v1/episodes/:id/publish`，走 `PATCH /v1/episodes/:id` 会静默忽略

写完 API 集成，一定要回读数据验证字段真的变了。

## 第二阶段：标题格式迁移（EPxxx_ → Exxx. ）

### 发生了什么

原格式 `EP444_清华教授：...` 有两个问题：下划线分隔不友好，`EP` 前缀与 `E` 不一致。改为 `E444. 清华教授：...`。

修改了 `upload_to_transistor_v2.py` 的标题生成逻辑后，发现一个隐藏问题：`reorder_episodes_by_date.py` 只更新 Transistor 的 `number` 字段，不更新 title 文本中的前缀。过去曾经重排过 EP 号，导致 233 个 episode 的 title 前缀跟实际 EP 号对不上（`number=441` 但 title 写的是 `EP444_`）。

做了两件事：
1. 批量把 450 个 title 从 `EP{N}_` 格式迁移到 `E{N}. ` 格式
2. 同步修正了 233 个错位的前缀数字

### 学到了什么

**冗余存储必然漂移。** EP 号同时存在两处：Transistor 的 `number` 字段 + title 文本里的前缀。任何只更新一处的操作都会造成不一致。解法要么消除冗余，要么每次更新都同步两处，不能靠约定。

**上传顺序必须在代码里强制。** 早期上传没有强制日期升序，导致最新视频拿到最小 EP 号。`sort(key=date, ascending=True)` 不能只是文档里的一句话。

## 第三阶段：文档重写

### 发生了什么

项目有一个 14K 字的 README，大量内容是历史操作日志和过时说明。还有一份自写的 `Transistor_API操作指南.md`，内容可能是幻觉或过时（Transistor 官方文档是权威）。

做了：
- README 压缩到 ~40 行：核心目标 + 项目结构 + 4 步快速命令 + 指向详细文档
- 重写 `核心任务说明.md`：4 步工作流、每步说明、archive 结构、API 限制、标题格式、工具目录
- 删掉 `Transistor_API操作指南.md`，替换为官方文档链接

### 学到了什么

**自写 API 手册是技术债。** 官方 API 一旦更新，自写手册就过时了。参考文档应该指向官方来源，自己的文档只写「在这个项目里怎么用」。

**文档太长等于没有文档。** 14K 字的 README 没人读完，关键信息被淹没。文档的目的是帮人快速找到需要的东西，而不是记录历史。

## 第四阶段：对账方式重构

### 发生了什么

项目用 `assets/summaries/视频与Transistor同步状态.csv` 做对账，但这个文件上传完之后从未被更新，实际上是过期数据。

重新思考后，用两个实时查询脚本替代：

- **`check_upload_candidates.py`**（日常用）：只看「比最新已发布内容更新」的待上传视频，有日期过滤，防止历史漏网视频混入
- **`check_transistor_sync.py`**（全量审计用）：无日期过滤，扫描所有本地视频，找出历史上所有从未上传的内容

两个脚本都直接查 API + 本地文件系统，终端打印结果，**零文件输出**。

### 学到了什么

**需要查的状态，就去查；不要缓存成文件。** 「现在有多少待上传视频」这个问题的答案永远在源头，不在上次生成的 CSV 里。文件天然腐烂，而且它的存在会让人以为它是最新的。

**日常工具和审计工具的需求不同。** 日常用需要过滤（只看新的），避免误操作；审计用需要完整视图，找历史漏洞。不要用同一个工具试图满足两种需求。

## 第五阶段：项目结构重组

### 发生了什么

逐一审查每个脚本的归属：

**脚本放错目录：**
- `check_upload_candidates.py`（只读）和 `find_duplicates.py`（只读）在 `tools/upload/`
  → 移入 `tools/check/`

**目录语义混乱：**
- `tools/generate/` 里混放了「生成报告」和「Whisper 转录」两类完全不同的东西
  → 转录脚本移入新建的 `tools/transcribe/`；报告生成脚本删除

**孤儿脚本：**
- `generate_summary.py`：为已废弃的「手动 review 下载进度」工作流服务
- `generate_video_documents.py`：为已转方向的内容写作工作流服务
- 两者均无消费者
  → 全部删除，连同输出产物（475 个 `视频文档/*.md`、`youtube_all_videos.txt`）

**孤儿符号链接：**
- `assets/videos` 指向项目外的 `../课代表立正_Archive`，12 月创建，来历不明
  → 删掉整个 `assets/` 目录

**`.gitignore` 漏洞：**
- `下载总结.md`、`数据完整性报告.md`、`duplicates_report.*` 等生成产物未被忽略
  → 全部补入 `.gitignore`

**最终结构：**
```
tools/
├── download/   1 个脚本（下载）
├── organize/   1 个脚本（分类）
├── transcribe/ 2 个脚本（Whisper 转录）
├── check/      5 个脚本（只读：对账、诊断）
└── upload/     3 个脚本（写远端：上传、排序、修复）
```

### 学到了什么

**目录名是对里面所有内容的承诺。** `tools/upload/` 里的脚本应该全部会写 Transistor，不然看到目录名就有错误的心智模型。按操作性质（读/写）划分比按使用场景划分更稳定。

**工作流重构时要同步清理工具。** 一个不再被任何流程调用的脚本，维护成本大于价值。判断标准：这个脚本上次被运行是什么时候？被谁调用？答不出来就是孤儿，直接删。

## 第六阶段：上传 GitHub

### 发生了什么

三个脚本（`upload_to_transistor_v2.py`、`reorder_episodes_by_date.py`、`fix_episode_metadata.py`）里硬编码了 Transistor API key。`核心任务说明.md` 的 curl 示例里也有明文 key。

做了：
1. `TRANSISTOR_API_KEY` 和 `TRANSISTOR_SHOW_ID` 移入 `.env`
2. 三个脚本改为内联 `.env` loader（无需 `python-dotenv` 依赖）
3. 文档 curl 示例换成 `YOUR_TRANSISTOR_API_KEY` 占位符
4. 加 `.env.example` 标注需要哪些环境变量
5. `uploaded_episodes.json` 补入 `.gitignore`
6. 创建 repo：[sunyuzheng/kedaibiao-content-tools](https://github.com/sunyuzheng/kedaibiao-content-tools)

### 学到了什么

**凡是打算公开的代码，上线前要做一次 secret 扫描。** `grep` 搜索所有已知 key 的片段，包括文档、注释、示例命令。API key 不只出现在 `.py` 文件里。

**`.env.example` 是必须的，不是可选的。** 克隆 repo 的人需要知道有哪些环境变量，填什么。`.env` 本身不进 git，但 `.env.example` 要进。

## 七条原则（浓缩）

| # | 原则 |
|---|------|
| 1 | 需要查的状态，就去查；不要缓存成文件 |
| 2 | 目录名是合约，按操作性质（读/写）划分 |
| 3 | 没有消费者的脚本是负债，工作流重构时同步清理 |
| 4 | 可重新生成的文件不进版本控制 |
| 5 | 文件移动和文档更新是同一个操作 |
| 6 | API 集成要显式验证副作用，不能信任 200 OK |
| 7 | 有序性依赖在代码层强制，不靠约定 |
