# 字幕校对工程复盘

> 时间：2026-04
> 目标：新录制视频 → Qwen3-ASR 转录 → Claude 校对 → 可用字幕
> 最终结果：v7，CER 提升 +1.50%，零回归，Precision 91.2%

---

## 背景

「课代表立正」的视频需要字幕用于播客和存档。Qwen3-ASR（Apple Silicon 本地模型）转录质量已经不错，但存在两类系统性错误：

1. **同音混淆**：沉浮/臣服、麦克/卖课、亚哥/鸭哥（嘉宾名）
2. **格式问题**：百分之十、两百、两千等中文数字未转为阿拉伯数字

目标是用 Claude 做后处理校对，但要保持保守（宁可不改也不乱改）。

---

## 架构

```
新视频
  └─ Qwen3-ASR（本地，Apple Silicon）
       └─ <stem>.qwen.srt
            └─ correct_srt.py（Claude）
                 ├─ 扫描候选词 → flags
                 ├─ LLM 逐 flag 表态（KEEP / 修改）
                 ├─ 验证层（拒绝幻觉 + 过量修改）
                 └─ <stem>.corrected.srt
```

**核心设计原则：保守优先。**
LLM 不做自由发挥，只对已知混淆模式（`correction_candidates.json`）的位置表态，额外发现的错误可以建议但受严格验证约束。

## Eval 框架

### CER（Character Error Rate）

```
CER = editDistance(corrected, human) / len(human)
```

以人工字幕为参考，测量校对后 SRT 与人工字幕的字符距离变化。
- **CER delta = CER_after - CER_before**（负数 = 更差，正数 = 更好）
- 测试集：29 个有人工字幕的视频（`archive/有人工字幕/` 中 20260101 之后的）

注意：**人工字幕不是绝对真值**，而是高精度参考。尤其是：
- 专有名词、嘉宾名：人工字幕偶有记音而非记字
- 语气词（嘛/吗）：人工字幕和转录可能不一致，这类差异不重要

### Precision / Recall（逐条修改质量）

对每条修改 `original → corrected` 和人工字幕比较：
- **GOOD**：corrected 在人工字幕，original 不在（真正修复了错误）
- **BAD**：original 在人工字幕，corrected 不在（破坏了正确内容）
- **NEUTRAL**：两者都在或都不在（无法判断）

**Precision = GOOD / (GOOD + BAD)**，不含 NEUTRAL（因为人工字幕和 Qwen 都可能是"另一种正确写法"）

### 候选词 Recall

对 `correction_candidates.json` 里每个模式：
- 出现在几个视频的 qwen.srt 里？
- 其中几个在 corrected.srt 里被正确处理？
- **Recall = 被处理 / 出现的视频数**

## 迭代历史

### 关键 Bug #1：格式规范化被编辑距离完全封死

**现象**：百分之百、两百、百分之十 recall 全部 0%。

**根因**：`_edit_distance_approx` 函数在验证层给每条修改设了阈值 ≤ 3。但中文 → 阿拉伯数字：
- `百分之十` → `10%`：edit_distance = 7（超限）
- `两百` → `200`：edit_distance = 5（超限）
- `百分之百` → `100%`：edit_distance = 8（超限）

所有格式规范化修改在验证层被静默丢弃。

**修复**：引入 `_FORMAT_NORM_PATTERNS` 集合，这些模式豁免编辑距离检查。同时把通用阈值从 3 提升到 4（允许 2 字同音对如 `它它→他他`）。

```python
_FORMAT_NORM_PATTERNS = {"百分之十", "百分之百", "两百", "两千", "幺幺", "到十"}

if orig not in _FORMAT_NORM_PATTERNS and _edit_distance_approx(orig, corr) > 4:
    continue  # 拒绝
```

**效果**：格式规范化 recall 从 0% 跳到 71-100%。

---

### 关键 Bug #2：`two → 2` 造成英文内容回归

**现象**：Je7Y0G6Qe8k（英文较多的视频）CER 下降 -1.41%。

**根因**：candidates.json 里有 `"two": {"alternatives": ["2"]}`，导致 "these two things"、"two column" 等英文短语中的 `two` 被替换为 `2`。

**修复**：删除 `two` 这条 candidate。`两` 字已有 `两百`/`两千` 两条更精确的规则覆盖中文语境。

---

### 关键 Bug #3：`就就 → 这这` 应用了错误的修正方向

**现象**：BAD corrections 样例中出现 `这这` 替换了 `就就`，但人工字幕里写的是 `就`（去掉重复）。

**根因**：candidates.json 里 `就就` 的 alternatives 是 `["这这"]`（同音混淆方向），但测试集里的 `就就` 实际上是口误重复，正确修法是删一个 `就`，而不是改成 `这这`。

**修复**：删除 `就就` 这条 candidate。这类口误重复不适合用候选词典处理（方向不确定）。

---

### 关键 Bug #4：`它它 → 他他` 改变了对象指代语义

**现象**：BAD corrections 中出现 `它` → `他` 用于指代代码/方法，语义完全错误。

**根因**：candidates.json 里有 `它它: {"alternatives": ["他他"]}`，但在技术内容中 `它` 指的是对象/函数，不是人。

**修复**：删除 `它它` 这条 candidate。

## 最终结果（v7）

| 指标 | 值 |
|------|-----|
| 测试集规模 | 29 个视频 |
| CER 平均提升 | **+1.50%** |
| 回归（CER 下降）| **0 / 29**（0%） |
| 有改动的视频 | 21 / 29（72%）|
| 总修改条数 | 81 条 |
| Precision | **91.2%** |
| 每视频 Precision 中位数 | 100% |

**候选词 Recall（主要词）：**

| 词 | Recall |
|----|--------|
| 亚哥→鸭哥 | **100%**（7/7 视频）|
| 百分之百→100% | **100%**（2/2 视频）|
| 两千→2000 | **89%** |
| 百分之十→10% | **71%** |
| 两百→200 | **44%** |
| 沉浮→臣服 | 测试集中 recall 低（触发频率不高）|

**已知无法解决的 BAD 样例（接受）：**
- `suae5Q`：亚→鸭 BAD，因为该视频的人工字幕本身也写的是「亚哥」（人工字幕的笔误，不是系统问题）
- `EfEk4V3`：志成→智沉 BAD，视频里的「赵志成」是不同人，而候选词针对的是「智沉」这位嘉宾

## 系统边界与局限

**不能解决的问题：**
1. **新嘉宾名字**：第一次出现的人名，系统性无法修正，需要手动加入 candidates.json
2. **上下文歧义词**：如 `两百` 出现在「一两百」里（模糊量词），boundary guard 尽力但不完美
3. **人工字幕与转录的非错误性差异**：语气词（嘛/吗）、口语化表达，这类差异不应被视为错误

**维护入口：**
- 加新候选词 → `logs/correction_candidates.json`
- 重新跑评估 → `python3 tools/correct/precision_eval.py`
- 跑批量校对 → `python3 tools/correct/batch_correct_srt.py`

## 关键 Learning（可迁移）

### 1. 验证层的静默丢弃是最难发现的 bug

系统没有报错，功能看起来"工作着"，但实际上所有格式修改都被悄悄抛弃了。

**对策**：对核心路径加 debug log，定期 sample 输出对比输入，确保改动确实发生了。

### 2. 候选词的粒度越精确，副作用越小

`two → 2` 这个规则想法正确（中文语境里 `两` 应该是数字），但颗粒度太粗，误伤英文。
正确的做法是拆成 `两百 → 200`、`两千 → 2000` 两条有明确上下文语义的规则。

**对策**：候选词写的是「触发 pattern」，不是「要替换的所有语义类」。宁可少一条规则，不要一条模糊规则。

### 3. 口误 vs 同音混淆需要区别处理

`就就` 重复出现是口误（应删一个），和 `亚哥/鸭哥` 的同音混淆是完全不同的错误类型。用同一套词典机制处理，方向可能相反。

**对策**：候选词典只处理「LLM 能确定替换方向的」情况。对于删除/合并类口误，考虑单独规则或不处理。

### 4. Human SRT 是高精度参考，不是绝对真值

用 human SRT 做 eval 参考时，需要接受它本身也有错误（人工字幕有笔误、有不同转写习惯）。
Precision 因此用「GOOD / (GOOD+BAD)」而非「GOOD / total」，NEUTRAL 不参与计算，避免被模糊情况稀释。

### 5. 越保守的 LLM 策略，越容易调试

系统的保守设计（LLM 只能对 flags 表态，不能自由发挥）让 bad cases 几乎全部可追溯到候选词典的条目，而不是 LLM 的随机行为。维护成本极低：只需维护 candidates.json，不需要改 prompt。

### 6. 评估要分层，不要只看一个指标

CER 改善但 Precision 低 → 有很多 BAD 改动，只是碰巧总体 CER 还是正的。
Precision 高但 CER 改善小 → 改动太少（Recall 低），需要加更多候选词。
候选词 Recall 低 → 验证层在悄悄拦截，或 LLM 在悄悄 KEEP。

**三个维度要同时看：CER delta + Precision + 候选词 Recall。**
